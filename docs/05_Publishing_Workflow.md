# ETLH Publishing Workflow

## Status

Canonical workflow for publishing and updating website articles in the current TypeScript-based application.

## Current source of truth

- `src/lib/posts.ts` — categories, article metadata, table-of-contents entries, and body blocks
- `src/assets/` — imported article images
- `src/routes/blog.$slug.tsx` — article rendering, metadata, responsibility signals, and Article structured data
- `src/lib/sitemap.ts` and `src/routes/sitemap[.]xml.ts` — public sitemap inventory

A planning document, social package, or draft file does not publish an article. Publication occurs only after the application source is updated, required validation passes, the change is merged, and production is verified.

## Required commands

```bash
bun install --frozen-lockfile
bun run check:ci
```

`check:ci` currently runs:

```bash
bun run lint && bun run check:sitemap && bun run check:content && bun run check:assets && bun run build
```

Formatting remains separate:

```bash
bun run format
```

Do not run repository-wide formatting as part of an unrelated publishing change.

## Article data requirements

Each post in `src/lib/posts.ts` requires:

- unique lowercase hyphenated `slug`
- final `title` and `excerpt`
- existing `category`
- verified `date` in `YYYY-MM-DD` format
- `readingTime`
- imported `image`
- useful `imageAlt`
- at least one tag
- `toc`
- non-empty `body`

Supported body blocks are `p`, `h2`, `h3`, and `ul`. Every TOC ID must match an `h2` or `h3` body ID.

Changing a published slug is a URL migration and must not be treated as a routine copy edit.

## Publication-date rule

Store the verified publication date directly in the post record. Do not infer it from file timestamps, draft creation, images, or conversation dates. Do not add a second date-override layer.

## Image rule

Add approved images to `src/assets/`, import them in `src/lib/posts.ts`, and provide accurate alt text.

All source images must pass `bun run check:assets`:

- maximum individual image: 400 KiB
- maximum total source-image library: 3 MiB

Use WebP or another appropriately compressed web format when a source image would exceed the budget. Preserve intrinsic dimensions and avoid misleading text, logos, or claims in imagery.

## Standard publishing procedure

1. Confirm the topic, wording, claims, evidence, and publication intent are approved under the Editorial and Research manuals.
2. Start from current `main` and create a focused branch.
3. Add or confirm the approved image asset.
4. Add one complete post record to `src/lib/posts.ts`.
5. Confirm slug, category, date, tags, TOC IDs, headings, image, and alt text.
6. Review the metadata generated by the shared article route. Avoid special-case metadata unless separately justified.
7. Run the frozen install and `bun run check:ci`.
8. Inspect the diff for unrelated changes, lockfile drift, unsupported claims, placeholder links, and generated output.
9. Open a focused pull request stating purpose, files, date source, category, image status, risks, tests, and rollback.
10. Confirm GitHub Actions passes and inspect the authoritative `eco-tiny-living-site` preview when available.
11. Merge only after acceptance gates pass.
12. Confirm the production deployment and record the merge commit and Vercel deployment ID.

## Preview acceptance

For a new or materially updated article, verify:

- article route returns 200
- title, excerpt, date, body, and image are correct
- image alt text is present
- TOC links reach the intended headings
- canonical and `og:url` use the preferred production hostname
- Open Graph and Twitter images use absolute URLs
- Article JSON-LD uses absolute page and image URLs
- ETLH responsibility and correction links remain visible
- footer and navigation remain intact
- an unknown route still returns a genuine 404

## Production acceptance

After deployment, confirm:

- the article route returns 200
- the article appears in `/blog`
- the article appears on its category page
- the article appears in `/sitemap.xml`
- canonical and structured-data URLs use `https://ecotinylivinghub.thrwds.com`
- production smoke checks pass

Do not mark an article Published based only on a draft, PR, or merge. Published status requires confirmation that the intended public release occurred.

## Generated route tree

Ordinary article and category data changes must not alter `src/routeTree.gen.ts`.

Route-file additions, removals, and renames should regenerate it. Never edit it manually. Follow `docs/08_Generated_Files_Policy.md` whenever a publishing change also adds or changes a public route.

## Updating an existing article

- preserve the slug unless a migration is explicitly approved
- preserve the original publication date unless a separate reviewed or updated date is intentionally added
- update claims and evidence under the Editorial and Research manuals
- rerun the full CI gate
- verify rendering and metadata when title, excerpt, headings, image, date, or schema changes
- record material corrections transparently

## Rollback

Revert the merge commit that introduced the article or update. Then rerun CI, confirm the reverted state deploys, verify the route/blog/category/sitemap behavior, and document why the rollback occurred.

## Current limitations

- Articles remain in one TypeScript file.
- Structural validation does not prove factual truth or editorial quality.
- Real-user Core Web Vitals and Search Console indexing require external traffic and owner-side evidence.
- A one-file-per-post system or CMS should be evaluated only after concrete publishing friction is documented.
